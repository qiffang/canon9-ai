# internal/storage — Data Persistence Layer

## Overview
Defines the `Store` interface (15 methods) and two filesystem-backed implementations: `FS` (read-write runtime store) and `BundleFS` (read-only bundle consumer). All agent tool operations flow through this layer.

## Structure
```
storage/
├── store.go          Store interface (15 methods)
├── event.go          Event struct + GenerateEventID()
├── sidecar.go        Sidecar, WikiPage, SearchResult, MemoryStats, EventsPage
├── fs.go             FS: read-write filesystem store (JSONL + Markdown + JSON sidecars)
├── bundle.go         BundleFS: read-only over a static bundle directory
├── fs_test.go        Filesystem store tests
```

## Store Interface

Single abstraction for all data operations: `AppendEvent`, `ReadEventsSince`, `ReadRecentEvents`, `ReadWikiIndex`, `ReadWikiPage`, `SearchWiki`, `WriteWikiPage`, `WriteWikiPageWithMeta`, `ArchiveWikiPage`, `RebuildIndex`, `SetCompileCursor`, `GetCompileCursor`, `GetMemoryStats`.

## Two Implementations

| Impl | File | Mode | Creates dirs? | Write methods? |
|------|------|------|---------------|----------------|
| `FS` | fs.go | Read-write | Yes (raw/, wiki/, wiki/.meta/) | Full |
| `BundleFS` | bundle.go | Read-only | No | All return errors |

- **`FS`** is used by the HTTP server and MCP runtime mode. It manages the full `data/` directory lifecycle.
- **`BundleFS`** is used by the MCP bundle mode and validator. It treats a directory as a complete, read-only OKF bundle.

## Filesystem Layout (FS mode)

```
data/
  raw/events.jsonl        Append-only JSONL event log
  raw/cursor              Decimal number (compile progress)
  wiki/index.md           Auto-generated routing table
  wiki/.meta/.../*.json   Sidecar per-page (provenance, access stats)
  wiki/semantic/          Concepts, people, projects, decisions
  wiki/episodic/          Contextual experiences
  wiki/procedural/        How-to, runbooks
  wiki/prospective/       Future intentions
  wiki/archive/           Deprioritized pages
```

## Sidecar Metadata

- **Separate from content**: Access telemetry in `.meta/*.json` changes frequently; wiki `.md` content changes rarely. Keeping them separate means git diffs are clean.
- **Per-page fields**: `CreatedAt`, `LastAccessed`, `AccessDates[]`, `SourceEvents[]`, `TrustTierMax`, `MemoryType`, `ArchivedAt`, `ArchiveReason`.
- **Writeback is best-effort**: Sidecar write failures are intentionally silenced (`_ = f.writeSidecar(...)`) to avoid blocking content operations.

## Event ID Format

`evt_YYYYMMDD_HHMMSS_rrrr` — generated by `GenerateEventID()` using `crypto/rand`. The 4 random chars provide ~16^4 = 65K possible IDs per second.

## Testing

- Always `t.TempDir()` — never hardcode paths. See fs_test.go.
- `FS` tests validate index rebuild, sidecar writeback, archive flow, wiki path validation.
- `BundleFS` is tested indirectly through MCP server tests and the validator.

## Gotchas

- **`AppendEvent` is thread-safe** (mutex-protected). Other operations (ReadWikiPage, WriteWikiPage) are NOT synchronized — callers must coordinate.
- **Sidecar write errors are silently discarded** in production (fs.go:212, 319, 357, 358). This is intentional for availability but means telemetry loss goes unnoticed.
- **`Rel` path failures are silenced** (fs.go:241, 389, 447; bundle.go:83, 133). A `filepath.Rel` error returns empty string, causing downstream issues.
- **Compile cursor is a simple uint64** stored as decimal in `raw/cursor`. No atomic increments — the compile agent's mutex provides serialization.
